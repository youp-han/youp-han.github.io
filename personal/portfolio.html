    Spring Boot & JPA 기반 사용자 통합 관리 API 서비스 개발/운영
    📌 개발 배경

    기존 고객사는 브랜드별로 서로 다른 솔루션을 운영하고 있었고, 각 솔루션이 개별적으로 사용자 정보를 관리하고 있었다.이로 인해 다음과 같은 문제점이 있었다.

    • 서비스별 중복 사용자 정보 발생
    • 최신 사용자 정보를 파악하기 어려움
    • Nets SSO 솔루션 사용으로 운영 비용 증가
    • SSO·AD 연동 과정에서 오류 관리가 중앙에 집중됨

    이 문제를 해결하기 위해 중앙집중형 사용자 관리 API를 Spring Boot 기반으로 구축하여 Keycloak을 중심으로 사용자 정보를 통합 관리하도록 목표를 설정했다.

    📌 개발 및 개선 내용
    1. 사용자 통합 API 구축
      • Spring Boot + JPA 기반으로 사용자 등록/수정/조회 API 개발
      • 각 서비스는 신규 API를 통해 사용자 데이터를 전달
      • API로 처리된 결과는 Keycloak–AD 간 Sync 프로세스를 통해 반영
      • 서비스별로 필요한 로컬 사용자 데이터는 Keycloak Attribute 기반으로 관리하도록 구조 설계
    2. 오류 분산 처리
      • 기존에는 AD에서 모든 데이터 오류가 집중적으로 발생
      • API 호출 서비스별로 오류를 분산하여 관리하도록 구조 변경
      • 데이터 동기화 오류는 호출한 서비스로 전달되어 정확한 근본 원인 파악 가능
    3. 데이터 Sync 구조 개선
      • Full Sync(스냅샷), Delta Sync(변경분) 방식 모두 지원
      • API 호출 즉시 반영되지 않더라도 Keycloak 기준으로 정확한 사용자 데이터 일관성 유지
    📌 문제 발생

    오픈 이후 타 브랜드들이 추가 연동되며 사용자가 약 30,000명 규모로 증가하였다.

    • Full Sync 수행 시간: 기존 약 15분 → 증가 후 매우 장시간 소요
    • API 호출 측에서 Request Timeout 발생
    • 실제 데이터 업데이트는 모두 정상 처리되었으나, 호출 측에서는 이를 확인하지 못해 오류로 분류됨
    • 대규모 사용자 Sync 로직의 비효율성이 병목 원인으로 분석됨
    📌 해결 방안
    1. API 전체 리팩토링

      초기 3개월 동안 급하게 개발된 코드를 전체 분석하고 구조를 재정비하였다.

      • 불필요한 조회/반복 처리 제거
      • JPA N+1 문제 제거
      • Batch 단위 처리 및 Stream 기반 처리로 성능 개선
    2. Sync 전략 분리(Full vs Partial)
      • Partial Sync(시간 단위 변경분 처리): 매 시간마다 변경된 사용자만 업데이트
      • Full Sync(하루 1회): 전체 데이터를 비교하여 반영
      • Full Sync 시 모든 데이터를 로딩하지 않도록 사용자 식별자만 저장하는 별도 테이블을 운영하여 비교 비용 절감
      • Sync 누락 가능성에 대비해, Sync 이후 생성된 신규 데이터는 자동 분류 및 별도 반영 로직 구현
    📌 결과
    • Full Sync 시간 대폭 단축 → API 호출 서비스에서도 결과 수신 가능
    • 저장/업데이트 과정에서 발생하던 구조적 오류 대부분 제거
    • 전체 사용자 데이터 관리가 통합되며 운영 효율성 상승
    • Keycloak 기반으로 SSO 및 AD 연동 관리가 안정화됨
    • 향후 신규 브랜드 추가 시 적용 부담 없이 확장 가능한 구조 마련
    SMB → AWS S3 파일 마이그레이션 프로그램 개발/운영
    📌 개발 배경

    기존에는 사내에서 SMB 기반 네트워크 드라이브(Windows Network Drive) 를 사용해 이미지 및 문서 파일을 저장하고 있었다. 하지만 다음과 같은 문제가 지속적으로 발생했다.

    • 노후화된 드라이브와 물리 디스크 장애로 데이터 손상 및 읽기 오류 증가
    • 대규모 파일 운영의 안정성 부족
    • 백업·이중화 어려움

    이를 개선하기 위해 네트워크 드라이브의 모든 파일을 AWS S3로 이전하여 안정적으로 관리하는 서비스를 구축하였다. 이를 위해 고객사가 보유한 대량의 파일(브랜드별로 수 TB, 최대 약 17TB)을 오류 없이 이전할 수 있는 전용 마이그레이션 프로그램 개발이 필요했다.

    📌 개발 내용

    초기 요구사항은 “S3로 파일을 업로드하는 마이그레이션 프로그램”이었으나, 실제 운영 과정에서 다음과 같은 부가 프로그램이 추가로 필요했다.

    • 메인 마이그레이션 프로그램 (.NET 기반)
      SMB 경로 탐색 → 파일 읽기 → DB 저장 → AWS S3 업로드 → S3 경로 DB 업데이트
      수십만 개 이상의 파일을 대상으로 전체 업로드 처리
    • 로컬 디렉토리 메타데이터 수집 프로그램
      하드 디스크의 전체 디렉토리 구조, 파일명, 경로, 파일 사이즈 등 메타데이터 수집
      대규모 마이그레이션 전 정확한 대상 파일 목록 생성
    • 오류 파일 재업로드 전용 프로그램
      기존 업로드 로그 분석
      오류·누락 파일 목록 정리
      재업로드 대상만 별도로 처리하는 프로그램

    이로써 단순 업로드 도구가 아니라, 대량 파일 마이그레이션을 위한 3단계 통합 시스템을 구축하게 되었다.

    📌 문제 발생

    메인 마이그레이션 프로그램은 다음의 순서로 동작했다:
    SMB 네트워크 드라이브에서 모든 파일을 순회하며 읽기
    파일 정보를 DB에 저장
    파일을 AWS S3에 업로드
    S3 경로를 DB에 다시 저장
    오류 발생 시 로그 기록 후 다음 파일 진행

    그러나 아래와 같은 문제가 반복적으로 발생했다.

    • 수십만 건의 파일 업로드 중 읽기 오류, 파일 권한 오류, 경로 깨짐, 파일 손상 등 다양한 예외 발생
    • 전체 업로드 중 오류 건수가 많아져 누락 파일 관리가 어려움
    • 모든 작업을 메인 프로그램 하나로 처리해 오류 관리/재시도 반복이 비효율적
    📌 해결 방안

    문제를 해결하기 위해 “오류 자동 점검·재업로드 구조”를 갖춘 프로그램을 추가 개발했다.

    • 1. 로컬 디렉토리 분석 프로그램 개발
      원본 디스크의 전체 파일 목록(경로/파일명/사이즈/수정일)을 스캔
      이를 기준으로 “업로드 대상 마스터 리스트” 생성
      기존 프로그램의 대상 경로 누락 문제 해결
    • 2. 오류 로그 수집 & 재업로드 파이프라인 구축
      마이그레이션 중 발생한 오류 로그 파일을 자동으로 분석
      오류로 처리된 파일들을 별도 DB/리스트로 정리
      재업로드 프로그램이 이 리스트만 읽어 누락 파일만 재처리하도록 구조 변경
    • 3. 메인 업로드 프로그램 기능 확장
      마스터 파일 리스트 또는 오류 파일 리스트만 선택해 업로드 가능하도록 기능 추가
      전체 마이그레이션 완료 시점에서 누락·오류 파일 없이 100% 반영 가능해짐
    📌 결과
    • 대량 파일 전송에서도 누락/오류 없이 안정적인 마이그레이션 달성
    • 최대 17TB 규모의 파일을 브랜드별로 성공적으로 AWS S3로 이전
    • 운영자가 오류 파일을 수작업으로 재확인하던 과정 완전 제거
    • SMB → S3 이전 후 서비스 운영 안정성 향상 및 유지보수 비용 절감
    ElectronJS 기반 데스크탑 애플리케이션 개발/운영
    📌 개발 배경

    기존 파일 관리 서비스는 웹 기반으로 개발되어 AWS 외부망에서 접근 가능하였다.
    하지만 고객사는 데이터 보안 정책상 내부망에서만 접근 가능해야 하는 기능이 추가되면서, 웹 환경만으로는 보안 요구사항을 충족할 수 없었다.
    고객사는 다음 조건을 충족하는 전용 데스크탑 애플리케이션 개발을 요청하였다.

    • 내부망에서만 동작해야 함
    • 실행 시 보안 프로그램이 구동되어 있어야 함
    • 웹 서비스와 동일한 기능을 제공하되, 외부망 접근을 원천적으로 차단할 것

    이 요구사항을 충족하기 위해 ElectronJS 기반 데스크탑 클라이언트를 개발하였다.

    📌 개발 내용
    • Electron 기반 데스크탑 앱 구현
      기존 웹 서비스를 Electron WebView 기반으로 통합
      로컬 환경에서만 접근 가능하도록 내부망 DNS/네트워크 정책과 연계
      UI/UX는 웹 버전과 동일하게 유지하여 사용자 전환 비용 최소화
    • 보안 프로그램 상태 연동
      고객사의 내부 보안 정책에 따라 데스크탑에서 다음을 수행하도록 구현하였다.
      애플리케이션 실행 시 보안 프로그램(사내 전용 보안 에이전트)의 실행 여부 검사
      보안 프로그램이 실행 중이 아니면 앱을 차단하고 사용자에게 경고 메시지 표시
      보안 프로그램이 정상 작동 중일 때만 파일 관리 기능 사용 가능하도록 제어
      이를 통해 기존 웹 버전에서 문제가 되었던 외부망 접근 및 인증 우회 문제를 구조적으로 차단할 수 있었다.
    📌 결과
    • 내부망 전용 파일 관리 서비스가 데스크탑 앱 형태로 안정적으로 제공됨
    • 보안 프로그램 연동으로 외부망 접근 차단 및 정보 유출 위험 제거
    • 사용자 환경은 웹과 동일하여 기존 사용자 교육/전환 비용 최소화
    • 사내 보안 준수 자동화로 운영 부서의 관리 부담 감소
    Zendesk API 기반 사용자 정보 수정 프로그램 개발
    📌 개발 배경

    통합 계정 서비스에서 정직원/파트너사 구분을 위해 이메일 체계가 변경되었다.
    이로 인해 기존에 파트너사들이 사용하던 Zendesk 고객센터 시스템에서 이메일 정보가 자동으로 동기화되지 않는 오류가 발생했다.
    문제는 다음과 같았다:

    • 통합 계정 서비스에서 이메일이 변경됨
    • 그러나 Zendesk는 내부적으로 이메일 변경을 자동 Sync 하지 못함
    • 그 결과, Zendesk 계정의 이메일이 실제 계정 정보와 불일치
    • 파트너사 문의 처리 및 인증 과정에 문제 발생

    Zendesk 자체 기능으로 해결할 수 없어, 모든 파트너사 계정을 강제로 업데이트하는 전용 수정 도구 개발이 필요해졌다.

    📌 개발 내용
    • Zendesk API 기반 계정 정보 수집
      Zendesk REST API를 이용해 전체 사용자 목록을 조회
      사용자 ID, 이메일, 태그, 그룹 등 메타데이터를 수집
      통합 계정 서비스에서 전달된 “최신 파트너사 이메일 목록”과 비교
    • 이메일 불일치 사용자 자동 검출
      변경 전·후 이메일 매칭 로직 구현
      기존 이메일과 업데이트된 파트너사 이메일이 불일치한 계정만 선별
      로그 기반으로 수정 필요 계정 리스트 자동 생성
    • 이메일 일괄 강제 업데이트 프로그램 개발
      Zendesk API의 사용자 Update 엔드포인트를 활용
      선별된 계정에 대해 이메일을 일괄 수정
      실패 시 재처리 리스트로 분류하여 안정성 강화
      .NET 기반 운영 툴 형태로 개발하여 관리자가 직접 실행 가능하도록 제공
    📌 문제 해결 결과
    • Zendesk 계정–통합 계정 간 이메일 불일치 문제 완전 해소
    • 모든 파트너사 계정의 이메일이 최신 정보 기준으로 정리됨
    • Zendesk 내 고객 정보/문의 처리의 정확성이 크게 향상
    • 반복적으로 발생하던 수동 수정 작업을 자동화하여 운영 부담 감소
    • 향후 이메일 정책 변경이 있어도 운영 툴로 신속하게 대응 가능
    VPN 사용자 관리 시스템 개발
    📌 개발 배경

    외부 네트워크 환경에서 내부 시스템 리소스를 안전하게 사용할 수 있도록 VPN을 새로 도입하게 되었다.
    이에 따라 다음 기능을 수행하는 운영자용 VPN 사용자 관리 프로그램이 필요해졌다.

    • VPN 사용자 등록 / 삭제
    • 사용자 일괄 업데이트
    • VPN 장비와 내부 시스템의 사용자 정보 Sync
    • 운영팀(Service Desk)이 직접 배치 처리 가능하도록 자동화

    이를 위해 VPN 장비 업체가 제공한 API 및 스펙을 분석해 전용 관리 시스템을 개발했다.

    📌 문제 1 — JSON 기반 업데이트 방식의 속도 문제

    초기 개발은 VPN 장비 업체에서 제공한 JSON 기반 사용자 업데이트 API 가이드를 그대로 적용하여 구현했다.
    하지만 테스트 과정에서 문제가 발견되었다.

    • 사용자 한 명 업데이트하는 데 약 1.5초 소요
    • 사용자 수가 수백~수천 명일 경우 총 소요 시간이 기하급수적으로 증가
    • 실제 운영에서 사용할 수 없는 수준으로 성능이 낮음
    ✔ 해결

    장비 업체와의 커뮤니케이션을 통해 JSON 방식이 아닌 XML 기반(XR 포맷) 업데이트 API가 존재함을 확인했다.
    이를 기반으로 프로그램을 수정한 결과:

    • 사용자 1명 업데이트 속도: 1.5초 → 0.5초
    • 기본 성능 약 3배 향상
    📌 문제 2 — 전체 데이터 재저장 방식에 따른 비효율

    초기 가이드에 따르면:

    • VPN 장비에는 기존 사용자 단건 조회 API가 없음
    • 그 결과, Sync를 위해 전체 사용자 정보를 매번 전체 업데이트해야 했음
    • 기존 사용자도 “추가/갱신” 방식으로 다시 저장
    • 삭제 작업까지 포함하면 전체 Batch 작업 시간이 매우 길어짐

    ⏱ 초기 성능
    약 2,000건 기준: 20분 이상 소요
    삭제 처리까지 포함하면 30분 이상
    운영자 입장에서 절대 사용할 수 없는 수준이었다.

    📌 개선 — 전체 사용자 조회 API 발견 및 Sync 로직 재설계

    오픈 이후 추가 개선 작업 시 다음 아이디어를 제안하고 구현했다.

    • 1. 장비 업체와 협력하여 전체 사용자 데이터를 포함한 다른 API를 확인
      기존 가이드에는 명시되지 않았지만, 실제로는 전체 VPN 장비 설정을 조회하는 API가 존재하고 있었음.
    • 2. 전체 데이터 중 “사용자 정보”만 필터링
      장비에서 내려받은 전체 데이터를 파싱
      사용자 리스트 부분만 추출
      이를 내부 시스템의 사용자 리스트와 비교
    • 3. 변경분(Delta) 만 업데이트하도록 구조 변경
      없는 데이터만 추가
      필요한 항목만 수정
      삭제도 비교 기반으로 수행
      전체 재저장 방식 제거
    ✔ 성능 결과
    • 전체 Sync 30분+ → 15초 미만으로 단축
    • 운영자가 실시간에 가깝게 사용자 동기화 가능
    • 불필요한 반복 업데이트 제거
    📌 결과
    • VPN 사용자 등록/삭제/Sync 자동화 프로그램 구축
    • 장비 스펙 제약을 우회하고 최적 성능을 만들 수 있는 구조로 자체 개선
    • 사용자 2,000명 기준 Sync 15초 이하로 처리 가능
    • Service Desk 운영팀의 업무 소요시간 획기적 감소
    • VPN 운영 안정성 및 사용자 경험 크게 향상
    프로젝트 협업 툴 및 CI/CD 파이프라인 관리
    📌 배경

    초기 환경에서는 Jenkins가 Master–Slave 구조로 구성되어 있었으며,
    운영 중인 대부분의 프로젝트가 .NET Framework 기반이어서 Windows 빌드 환경 위주로 운용되고 있었다.
    그러나 시간이 지나면서 다음과 같은 변화가 발생했다.

    • Java 기반 서비스 증가
    • Jira/Confluence 등 협업 툴과의 연동 요구
    • 모바일(Android/iOS) 빌드 환경 추가
    • 외부/내부 서비스와 Jenkins API 연동 작업 증가

    이에 따라 기존 단일 Jenkins Master에 모든 프로젝트가 몰리며 다음 문제가 발생했다.

    • 빌드 큐 병목
    • 서비스별 요구환경 충돌
    • 운영팀/개발팀의 Jenkins API 호출량 폭증
    • 장애 시 전체 서비스 영향도 증가

    고객사는 안정성을 위해 기존 단일 Jenkins 환경을 서비스별로 완전히 분리된 개별 Jenkins로 나누는 작업을 요청했다.

    📌 역할 및 수행 작업
    • 1. 서비스별 Jenkins 인프라 분리 설계 및 구축
      Windows, Linux, macOS 등 프로젝트 특성에 따라 다음을 개별 환경으로 구성했다.
      • .NET / .NET Framework 빌드용 Windows Jenkins
      • Java & Spring Boot 빌드용 Linux Jenkins
      • 모바일(Android/iOS) 빌드용 macOS Jenkins
      • 기타 스크립트/배치 전용 Jenkins

      서비스 운영자들과 지속적으로 소통하며, 각 서비스가 필요로 하는 빌드 도구·SDK·플러그인을 설치 및 최적화했다.
    • 2. Jenkins 파이프라인 코드(Jenkinsfile) 기반으로 CI/CD 구조 재정립
      프로젝트 수가 많고 UI 기반 설정은 관리가 어려웠기 때문에 Jenkins의 파이프라인 스크립트를 적극 도입했다.
      도입 효과:
      • CI/CD 설정을 코드(Jenkinsfile) 로 관리 → 버전 관리 가능
      • 유사한 환경의 프로젝트들은 코드 재사용 가능
      • 공통 로직(빌드/테스트/배포/Slack 알림/아티팩트 관리 등)을 모듈화
      • Jenkins UI 설정 오류 및 환경 차이를 원천 차단
      • 신규 서비스 온보딩 속도 단축
    • 3. 협업 툴 및 외부 시스템 연동
      • Jira 이슈 트래킹과 Jenkins 빌드를 연동하여 자동 상태 업데이트
      • 빌드/배포 결과를 Slack, Email로 자동 알림
      • 아티팩트 버전 관리 및 Confluence 문서 자동 업데이트
      • Jenkins REST API를 통한 외부 서비스 배포 자동화 구현
    📌 결과
    • 서비스별 Jenkins 분리로 빌드 안정성 대폭 향상
    • Master 장애가 전체 서비스에 영향을 주던 구조에서 → 독립 서비스 구조로 전환
    • 운영팀의 API 호출 과부하 문제 해소
    • Jenkinsfile 기반으로 CI/CD 관리 표준화 → 설정 재사용률 증가, 오류 감소
    • 신규 서비스 추가 시 설정 복잡도를 크게 감소
    • 개발팀/운영팀 모두 CI/CD 프로세스를 명확하게 이해하고 사용 가능
