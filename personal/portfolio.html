    📌 개발 배경

    기존 고객사는 브랜드별로 서로 다른 솔루션을 운영하고 있었고, 각 솔루션이 개별적으로 사용자 정보를 관리하고 있었음.이로 인해 다음과 같은 문제점 발생.

    • 서비스별 중복 사용자 정보 발생
    • 최신 사용자 정보를 파악하기 어려움
    • Nets SSO 솔루션 사용으로 운영 비용 증가
    • SSO·AD 연동 과정에서 오류 관리가 중앙에 집중됨

    이 문제를 해결하기 위해 중앙집중형 사용자 관리 API를 Spring Boot 기반으로 구축하여 Keycloak을 중심으로 사용자 정보를 통합 관리하도록 목표를 설정.

    📌 개발 및 개선 내용
    1. 사용자 통합 API 구축
      • Spring Boot + JPA 기반으로 사용자 등록/수정/조회 API 개발 - 본인 진행
      • 각 서비스는 신규 API를 통해 사용자 데이터를 전달
      • API로 처리된 결과는 동시 개발 진행 된 Keycloak–AD 간 Sync 프로세스를 통해 반영 - AA 팀 진행
      • 서비스별로 필요한 로컬 사용자 데이터는 Keycloak Attribute 기반으로 관리하도록 구조 설계
    2. 오류 분산 처리
      • 기존에는 AD에서 모든 데이터 오류가 집중적으로 발생
      • API 호출 서비스별로 오류를 분산하여 관리하도록 구조 변경
      • 데이터 동기화 오류는 호출한 서비스로 전달되어 원인 파악 후 수정 및 재호출
    3. 데이터 Sync 구조 개선
      • Full Sync(스냅샷), Delta Sync(변경분) 방식 지원
      • API 호출 즉시 반영, (5분 배치) Keycloak 기준으로 정확한 사용자 데이터 일관성 유지
    📌 문제 발생

    오픈 이후 다른 브랜드가 추가 연동되며 사용자가 약 8배 규모로 증가.

    • Full Sync 수행 시간: 첫 브랜드에서 약 15분 → 추가/증가 후 매우 장시간 소요
    • API 호출 측에서 Request Timeout 발생
    • 실제 데이터 업데이트는 모두 정상 처리되었으나, 호출 측에서는 이를 확인하지 못해 오류로 분류됨
    • 대규모 사용자 Sync 로직의 비효율성이 병목 원인으로 분석됨
    📌 해결 방안
    1. API 전체 리팩토링

      초기 3개월 동안 급하게 개발된 코드를 전체 분석하고 구조를 재정비.

      • 불필요한 조회/반복 처리 제거
      • Batch 단위 처리 및 저장 방식 변경으로 성능 개선
    2. Sync 전략 분리(Full vs Partial)
      • Partial Sync(시간 단위 변경분 처리): 매 시간마다 변경된 사용자만 업데이트
      • Full Sync(하루 1회): 전체 데이터를 비교하여 반영
      • Full Sync 시 모든 데이터를 로딩하지 않도록 사용자 식별자만 저장하는 별도 테이블을 운영하여 비교 소요시간 절감
      • Sync 누락 가능성에 대비해, Sync 이후 생성된 신규 데이터는 자동 분류 및 별도 반영 로직 구현
    📌 결과
    • Full Sync 시간 대폭 단축 → API 호출 서비스에서도 결과 수신 가능

    SMB → AWS S3 파일 마이그레이션 프로그램 개발/운영
    📌 개발 배경

    기존에는 사내에서 SMB 기반 네트워크 드라이브(Windows Network Drive) 를 사용해 이미지 및 문서 파일을 저장. 하지만 다음과 같은 문제가 지속적으로 발생.

    • 노후화된 드라이브와 물리 디스크 장애로 데이터 손상 및 읽기 오류 증가
    • 대규모 파일 운영의 안정성 부족
    • 백업·이중화 어려움

    AA팀에서 이를 개선하기 위해 네트워크 드라이브의 모든 파일을 AWS S3로 이전하여 안정적으로 관리하는 서비스 구축. 이를 위해 고객사의 여러 브랜드별 보유한 대량의 파일(브랜드별로 수 TB, 최대 약 17TB)을 오류 없이 이전할 수 있는 전용 마이그레이션 프로그램 개발이 필요.

    📌 개발 내용

    “S3로 파일을 업로드 및 다운로드 하는 마이그레이션 프로그램”

    • 메인 마이그레이션 프로그램 (.NET 기반)
      SMB 경로 탐색 → 파일 읽기 → DB 저장 → AWS S3 업로드 → S3 경로 DB 업데이트
      수십만 개 이상의 파일을 대상으로 전체 업로드 처리
    📌 문제 발생
    • 수십만 건의 파일 업로드 중 읽기 오류, 파일 권한 오류, 경로 깨짐, 파일 손상 등 다양한 예외 발생
    • 전체 업로드 중 오류 건수가 많아져 누락 파일 관리가 어려움
    • 모든 작업을 메인 프로그램 하나로 처리해 오류 관리/재시도 반복이 비효율적
    📌 해결 방안

    문제를 해결하기 위해 “오류 자동 점검·재업로드 구조”를 갖춘 프로그램을 추가로 개발.

    • 1. 로컬 디렉토리 분석 프로그램 개발
      파일서버 내 이동 할 디스크의 전체 파일 목록(경로/파일명/사이즈/수정일)을 스캔
      기존 프로그램의 대상 경로 누락을 대비
      “업로드 대상 마스터 리스트” 생성
    • 2. 오류 로그 분석 & 재업로드 를 위한 리스트 작성 프로그램 개발
      마이그레이션 중 발생한 오류 로그 파일을 자동으로 분석
      오류로 처리된 파일들을 별도 리스트로 정리
      재업로드를 위해 누락 파일을 원본 파일과 비교하여 재처리 되도록 리스트 작성
    • 3. 메인 업로드 프로그램 기능 확장
      추가로 오류 파일 리스트만 선택해 업로드 기능 개발
      전체 마이그레이션 완료 시점에서 누락·오류 파일 없이 100% 반영 가능해짐
    📌 결과
    • 대량 파일 전송에서도 누락/오류 없이 안정적인 마이그레이션 달성
    • 최대 10TB 이상 많은 파일들을 브랜드별로 성공적으로 AWS S3로 이전
    • 운영자가 오류 파일을 수작업으로 재확인하던 과정 완전 제거

    VPN 사용자 관리 시스템 개발
    📌 개발 배경

    외부 네트워크 환경에서 내부 시스템 리소스를 안전하게 사용할 수 있도록 VPN 장비 새로 도입함.
    이에 따라 사용자 관리를 수행하는 운영자용 VPN 사용자 관리 프로그램 을 보안팀에서 AA 팀으로 요청. 요청사항은 회의를 통해 정리되었고, 개발 진행함.

    • 요청에 의한 VPN 사용자 등록 / 삭제
    • 배치작업으로 통한 사용자 정보 일괄 업데이트
    • 운영팀(Service Desk)이 직접 작업 가능하도록 화면 추가

    이를 위해 VPN 장비 업체가 제공한 API 가이드 및 스펙을 분석해 전용 관리 시스템 개발 진행.

    📌 문제 1 — JSON 기반 업데이트 방식의 속도 문제

    초기 개발은 VPN 장비 업체에서 제공한 JSON 기반 사용자 업데이트 API 가이드를 그대로 적용하여 구현.
    하지만 테스트 과정에서 문제 발생.

    • 배치 작업 시, 사용자 한 명 업데이트하는 데 약 1.5초 소요
    • 배치 작업 시, 사용자 수 2000 명을 타겟으로 개발 중이여서, 총 소요 시간이 기하급수적으로 증가
    • 실제 운영에서 사용할 수 없는 수준으로 성능이 낮음
    ✔ 해결

    장비 업체와의 커뮤니케이션을 통해 JSON 방식이 아닌 XML 기반(XR 포맷) 업데이트 API 확인.
    이를 기반으로 프로그램을 수정한 결과:

    • 사용자 1명 업데이트 속도: 1.5초 → 0.5초
    • 기본 성능 약 3배 향상
    📌 문제 2 — 전체 데이터 재저장 방식에 따른 비효율

    초기 가이드에 따르면:

    • VPN 장비에는 사용자로 등록된 사용자들의 조회 API가 없음
    • 그 결과, 배치 시 전체 사용자 정보를 매번 모두 전체 업데이트해야 했음
    • 기존 사용자도 “추가/갱신” 방식으로 다시 저장
    • 삭제 작업까지 포함하면 전체 Batch 작업 시간이 매우 길어짐

    ⏱ 초기 성능
    약 2,000건 기준: 20분 이상 소요

    📌 배경

    초기 환경에서는 Jenkins가 Master–Slave 구조로 구성되어 있었으며,
    운영 중인 대부분의 프로젝트가 .NET Framework 기반이어서 Windows 빌드 환경 위주로 운용되고 있었음.
    그러나 시간이 지나면서 다음과 같은 변화가 발생.

    • Java 기반 서비스 증가
    • Jira/Confluence 등 협업 툴과의 연동 요구
    • 외부/내부 서비스와 Jenkins API 연동 작업 증가

    이에 따라 기존 단일 Jenkins Master에 모든 프로젝트가 몰리며 다음 문제가 발생했다.

    • 빌드 큐 병목
    • 장애 시 전체 서비스 영향도 증가

    AA팀에서, 배포 안정성을 위해 기존 단일 Jenkins 환경을 서비스별로 완전히 분리된 개별 Jenkins로 나누는 작업을 요청.

    📌 역할 및 수행 작업
    • 1. 서비스별 Jenkins 인프라 분리 설계 및 구축
      Windows, Linux 프로젝트 특성에 따라 다음을 개별 환경으로 구성.
      • .NET / .NET Framework 빌드용 Windows Jenkins
      • Java & Spring Boot 빌드용 Linux Jenkins
      • 기타 스크립트/배치 전용 Jenkins

      서비스 운영자들과 지속적으로 소통하며, 각 서비스가 필요로 하는 빌드 도구·SDK·플러그인을 설치 및 최적화 함.
    • 2. Jenkins Pipeline 코드 기반으로 CI/CD 구조 재정립
      AA팀에서 고객사의 프로젝트 수가 많고 UI 기반 설정은 관리가 어려웠기 때문에 Jenkins의 파이프라인 스크립트를 적극 도입토록 요청.
      도입 효과:
      • CI/CD 설정을 코드(Pipeline) 로 관리
      • 유사한 환경의 프로젝트들은 코드 재사용 가능
      • 공통 로직(빌드/테스트/배포/Slack 알림/아티팩트 관리 등)의 재사용 가능
    • 3. 협업 툴 및 외부 시스템 연동
      • Jira 이슈 트래킹과 Jenkins 빌드를 연동하여 자동 상태 업데이트
      • 빌드/배포 결과를 Slack, Email로 자동 알림
      • Nexus 에서의 아티팩트 버전 관리
      • Jenkins REST API를 통한 외부 서비스 에서 버전 별 배포 구현
